# Portfolio copy references

## Purpose

The portfolio copy should not sound sentimental, childish, or AI-generated. Avoid "따뜻함", "살펴본다", "함께합니다" style filler unless the source material actually supports it.

The writing should be grounded in real work:

- operating and changing existing systems;
- reading code, DB, API, server, and deployment flow together;
- carrying legacy maintenance experience into modern full-stack work;
- treating AI tools as workflow acceleration, not personality branding.

## References worth borrowing from

### Maggie Appleton — `https://maggieappleton.com/about`

What to borrow:

- Layered identity instead of a flat job title.
- Honest career framing: a career can sit between several disciplines and still be coherent.
- Specific history, not résumé bullet dumping.

Useful pattern:

> I sit at the intersection of X, Y, and Z. Combining them into a coherent career is an ongoing challenge.

Adapted for this portfolio:

> 제 경력은 레거시 유지보수, 업무 시스템 개발, AI 보조 개발 사이에 있습니다. 세 영역은 따로 보이지만, 실제로는 모두 같은 질문으로 이어집니다. 이미 돌아가는 시스템을 얼마나 정확히 읽고, 어디까지 안전하게 바꿀 수 있는가.

### Tom MacWright — `https://macwright.com/about/`

What to borrow:

- Concrete built-history.
- Directness: "Right now I'm building..." then what he built before.
- Avoid abstract adjectives. Let the list of real work carry weight.

Useful pattern:

> Right now I'm building X. Before that, I helped build Y, did Z, and worked on A.

Adapted for this portfolio:

> 지금은 AWP 업무 시스템과 3D/BIM 뷰어를 다루고 있습니다. 그 전에는 문자 발송 서버, 앱 API와 관리자 페이지, 쇼핑몰과 웹뷰 앱을 운영했습니다. 새 화면을 만드는 일도 했지만, 더 많이 한 일은 이미 돌아가는 서비스를 고치고, 연결하고, 오래 버티게 만드는 일이었습니다.

## Copy rules for this site

1. Do not write emotional warmth copy.
   - Bad: "누구에게 따뜻함이 필요한지 살펴봅니다."
   - Good: "운영 중인 시스템에서 어디가 막히는지 먼저 읽습니다."

2. Do not write AI-sounding value statements.
   - Bad: "기술과 사람을 연결하는 따뜻한 개발자입니다."
   - Good: "화면, API, DB, 서버 흐름을 함께 읽고, 변경 범위를 작게 잡아 고칩니다."

3. Use tension.
   - Legacy vs modern stack.
   - Speed vs verification.
   - AI automation vs developer judgment.
   - New feature vs existing service stability.

4. Use concrete nouns.
   - CodeIgniter, Spring Boot, AG Grid, Workpackage, BIM, xeokit, pgvector.

5. If a sentence could apply to any developer, delete it.

## New core copy direction

### Main line

이미 돌아가는 시스템을 읽고, 필요한 만큼 바꾸는 개발자입니다.

### Support copy

> 처음부터 새로 짜는 일보다 어려운 건, 이미 운영 중인 서비스 안에서 변경 가능한 선을 찾는 일입니다. PHP/CodeIgniter 쇼핑몰 유지보수에서 시작해 문자 발송 서버, 앱 API와 관리자 페이지, AWP 업무 시스템과 3D/BIM 뷰어까지 다뤘습니다. 화면을 고칠 때 API 응답, DB 구조, 권한, 서버와 배포 이후 영향을 같이 봅니다.

### AI line

> AI는 판단을 대신하지 않습니다. Cursor, Codex, OpenCode는 낯선 코드의 흐름을 따라가고, 구현안을 비교하고, 검증 목록을 반복하는 시간을 줄이는 도구입니다. 최종 책임은 여전히 사람이 져야 합니다.

## Second-pass tightening

After the first deeper rewrite, remove any sentence that still talks about the redesign process instead of the developer. In particular, avoid public phrases like `새 프로젝트를 처음부터 예쁘게 만드는 일보다`; that sounds defensive. Use the cleaner tension instead: `처음부터 새로 짜는 일보다 어려운 건, 이미 운영 중인 서비스 안에서 변경 가능한 선을 찾는 일입니다.`
