## Current recommended hero

### Main line

이미 돌아가는 시스템을 읽고, 필요한 만큼 바꾸는 개발자입니다.

### Support copy

새 프로젝트를 처음부터 예쁘게 만드는 일보다 더 많이 해온 것은, 이미 운영 중인 서비스 안에서 무엇이 깨졌고 어디까지 바꿀 수 있는지 읽어내는 일이었습니다. PHP/CodeIgniter 쇼핑몰 유지보수에서 시작해 Spring Boot API, React/TypeScript 화면, DB와 서버 운영, 배포 흐름까지 옮겨 다니며 한 기능이 실제 운영 환경에서 어떻게 살아남는지 배웠습니다.

### AI workflow line

AI는 판단을 대신하지 않습니다. 대신 낯선 코드를 읽고, 구현 초안을 만들고, 리팩터링 후보를 정리하고, 검증 체크리스트를 반복하는 시간을 줄여줍니다. Cursor, Codex, OpenCode, AGENTS.md, Skills, MCP를 쓰는 이유도 거기에 있습니다. 자동화는 일을 덜 생각하게 만드는 장치가 아니라, 더 중요한 판단에 시간을 남기는 장치여야 합니다.

## Short version

이미 돌아가는 시스템을 읽고, 필요한 만큼 바꾸는 개발자입니다. PHP/CodeIgniter 유지보수에서 시작해 Spring Boot API, React/TypeScript 화면, DB와 서버 운영까지 옮겨 다니며 운영 중인 서비스의 변경 범위를 읽는 법을 배웠습니다. AI 도구는 판단을 대신하는 장치가 아니라 반복을 줄여 판단할 시간을 남기는 장치로 사용합니다.

## Professional version

웹 서비스 유지보수와 운영 경험을 바탕으로 화면, API, 데이터베이스, 서버 흐름을 연결해 구현하는 풀스택 개발자입니다. AWP 업무 시스템과 3D/BIM 뷰어, 문자 발송 서버, 앱 API 및 관리자 페이지, 쇼핑몰/웹뷰 앱 유지보수처럼 이미 운영 중인 시스템을 다뤄왔습니다. 최근에는 Cursor, Codex, OpenCode, AGENTS.md, Skills, MCP를 활용해 코드 탐색, 구현 초안, 문서화, 반복 검증 시간을 줄이는 개발 흐름을 만들고 있습니다.

## Section function definitions

If final copy is not approved yet, use section placeholders only:

1. **소개** — 이미 운영 중인 시스템을 읽고 변경하는 개발자 정체성

3. **경력 기반 프로젝트** — AWP/BIM, 문자 발송 서버, 앱 API, 쇼핑몰 유지보수
4. **AI 활용 방식** — 코드 탐색, 구현 초안, 리팩터링, 문서화, 반복 검증 자동화
5. **기술 스택** — 기능별로 묶은 기술 목록

## Do not use

- “누구에게 따뜻함이 필요한지 살펴봅니다” 같은 감성 문장을 쓰지 않는다.
- “기술과 사람을 연결하는 따뜻한 개발자” 같은 누구에게나 붙는 문장을 쓰지 않는다.
- “AI에게 일을 맡기고…”를 메인 소개로 쓰지 않는다. 내부 작업 방식 설명에는 가능하지만, 이력서 기반 포트폴리오 첫 소개로는 추상적이다.
- “Loop Dog Lab”, “댕댕이/멍멍이”를 첫 화면의 주인공으로 쓰지 않는다.
- “귀여운 척을 빼고” 같은 제작 과정 메타 문구를 공개 페이지에 넣지 않는다.
- 검증되지 않은 사이드 프로젝트를 경력보다 먼저 내세우지 않는다.
